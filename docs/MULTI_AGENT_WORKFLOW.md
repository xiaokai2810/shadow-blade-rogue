# 多 Agent 游戏开发协作体系

本文定义 `影刃花海` 的多 Agent 协作方式。目标是让每次功能迭代都能从需求、架构、开发、测试、美术、Review 到再迭代形成闭环，并且适配当前项目的网页端、抖音小游戏端和 Android APK 端。

## 协作链路

```text
产品 Agent -> 架构 Agent -> 开发 Agent -> 测试 Agent -> 美术 Agent -> Review Agent -> 迭代
```

每个 Agent 只对自己的交付物负责，但必须给下游留下清晰输入。任何 Agent 发现上游目标不清、风险过高或验收条件不可测时，可以退回上游补充。

## 基本原则

- 先定义玩家价值，再定义技术实现。
- 每个任务必须有验收标准，不能只有想法。
- 网页端、抖音小游戏端、Android 端共享核心玩法逻辑，避免三份逻辑分叉。
- 新素材进入游戏前必须有文件名、尺寸、用途、回退方案。
- Review Agent 有权阻止发布，但必须给出可执行修改项。
- 所有密钥、服务器密码、GitHub token 只保存在本机 Keychain 或受限配置中，不进入仓库。

## Agent 职责

| Agent | 核心职责 | 主要产物 | 下游输入 |
| --- | --- | --- | --- |
| 产品 Agent | 定义玩家目标、版本范围、核心指标、玩法优先级 | 需求卡、验收标准、版本目标 | 架构设计 |
| 架构 Agent | 拆模块、定接口、评估多端兼容和数据结构 | 技术方案、模块边界、风险清单 | 开发任务 |
| 开发 Agent | 实现功能、保持代码风格、同步多端工程 | 代码、脚本、配置、变更说明 | 测试用例 |
| 测试 Agent | 验证功能、回归核心链路、检查多端风险 | 测试报告、缺陷单、复现步骤 | 修复任务或 Review |
| 美术 Agent | 维护风格、生成/整理素材、检查视觉一致性 | 资产清单、素材文件、视觉 QA | 开发整合或 Review |
| Review Agent | 代码审查、发布审查、性能和安全审查 | Review 结论、阻断项、发布建议 | 迭代决策 |
| 迭代协调 | 排期、合并结论、决定继续/发布/回滚 | 迭代记录、版本日志、下一轮任务 | 下一轮产品输入 |

## 标准交接包

每个 Agent 完成任务后，必须按下面格式交接：

```text
任务 ID:
当前阶段:
目标:
已完成:
关键文件:
验证方式:
风险/未决问题:
建议下一步:
```

当任务跨端影响较大时，额外补充：

```text
网页端影响:
抖音小游戏影响:
Android 影响:
发布影响:
```

## 工作流门禁

### Gate 1: 产品冻结

进入架构前必须明确：

- 玩家要获得什么体验提升。
- 成功指标是什么，例如首局时长、升级次数、死亡原因、Boss 击杀率。
- 本轮不做什么，避免范围漂移。
- 验收标准可以被测试 Agent 复现。

### Gate 2: 架构冻结

进入开发前必须明确：

- 修改哪些核心模块。
- 是否影响 `src/blade-survivor.js`、`douyin-minigame/src/blade-survivor.js`、`android-apk/app/src/main/assets/src/blade-survivor.js` 的同步。
- 是否需要新增资源加载、存档字段、平台 API。
- 回滚方案。

### Gate 3: 开发完成

进入测试前必须满足：

- 代码已实现并自查。
- 多端共享逻辑已同步。
- 没有把 `dist/`、APK build 产物、密钥、临时 QA 图提交进仓库。
- 至少通过 `npm run check`。

### Gate 4: 测试通过

进入美术/Review 前必须满足：

- 桌面键盘和移动虚拟摇杆至少各完成一局。
- 升级、Boss、死亡结算、重新开始都可用。
- 网页端静态构建通过。
- 如果改动平台适配，抖音小游戏结构检查通过。

### Gate 5: 美术通过

进入 Review 前必须满足：

- 新图片能加载，失败时有合理 fallback。
- 色彩和主角/怪物识别度符合当前风格。
- 移动屏幕下不遮挡关键 UI。
- 素材文件名、用途、尺寸记录到资产清单。

### Gate 6: 发布 Review

发布前必须满足：

- Review Agent 没有 P0/P1 阻断项。
- `npm run check` 通过。
- `npm run build:site` 通过。
- GitHub Pages 与服务器同步策略明确。
- 抖音小游戏审核资料不包含虚假资质、备案号或版号。

## 分支和任务规则

- 小改动可以直接在 `main` 工作，但必须经过 Gate 3 和 Gate 4。
- 大功能使用 `feature/<short-name>`。
- 每个 issue 只绑定一个主 Agent，其他 Agent 通过交接包补充。
- 一个功能最多拆成三个并行开发任务：玩法、平台适配、素材/UI。
- Review 只看风险和质量，不重新讨论已经冻结的产品目标，除非发现目标不可发布。

## Agent Prompt 模板

### 产品 Agent

```text
你是产品 Agent。请基于当前游戏状态，为本轮迭代定义玩家价值、范围、验收标准和不做项。
输出需求卡，不写代码。
必须包含：目标玩家体验、成功指标、核心规则、边界条件、验收标准。
```

### 架构 Agent

```text
你是架构 Agent。请把产品需求拆成可实现方案，评估网页端、抖音小游戏端、Android 端影响。
输出技术方案，不写业务代码。
必须包含：模块边界、数据结构、同步文件、风险、回滚方案。
```

### 开发 Agent

```text
你是开发 Agent。请按架构方案实现功能。
必须遵守现有代码风格，避免无关重构。
完成后输出：改动文件、验证方式、需要测试 Agent 关注的风险。
```

### 测试 Agent

```text
你是测试 Agent。请验证本轮功能和核心回归链路。
必须覆盖：启动、移动、攻击、升级、Boss、死亡、重开、资源加载、多端结构检查。
输出通过项、失败项、复现步骤和风险等级。
```

### 美术 Agent

```text
你是美术 Agent。请维护游戏风格并处理素材需求。
必须输出：素材清单、尺寸、命名、用途、预览检查、加载失败 fallback 建议。
```

### Review Agent

```text
你是 Review Agent。请按代码审查方式输出问题。
 findings 优先，按严重程度排序，包含文件和行号。
关注：玩法回归、多端分叉、性能、资源路径、发布风险、密钥泄漏。
```

## 迭代节奏

一个标准迭代最多 3 天：

| 时间 | 内容 |
| --- | --- |
| Day 1 上午 | 产品 Agent 冻结需求 |
| Day 1 下午 | 架构 Agent 出方案，Review Agent 预审风险 |
| Day 2 | 开发 Agent 实现，美术 Agent 同步素材 |
| Day 3 上午 | 测试 Agent 回归 |
| Day 3 下午 | Review Agent 发布审查，迭代协调决定发布或返工 |

紧急修复可走快速链路：

```text
测试 Agent 复现 -> 开发 Agent 修复 -> Review Agent 确认 -> 发布
```

## 应用于当前游戏

当前游戏已经具备核心闭环：移动、自动攻击、升级、Boss、结算、网页发布、服务器发布、GitHub Pages 发布。下一阶段多 Agent 协作重点应该放在三件事：

1. 提高局内成长深度：武器流派、遗物、局外成长。
2. 提高内容识别度：怪物阵营、Boss 技能、主角动作反馈。
3. 提高发布稳定性：网页、抖音小游戏、Android 三端同步检查自动化。

具体任务池见 `docs/GAME_AGENT_BACKLOG.md`。
